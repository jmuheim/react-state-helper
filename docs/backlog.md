## Lighten the script: move the default state JSON out of the code

The 600+ line script appears to be a real performance burden on MobileCoach. The simplest first step: the app content data (`defaultStateTemplate()`, roughly 170 of the ~690 lines in `src/ReactStateHelper.js`) is pure JSON and doesn't need to live in the script at all. Move it to its own JSON file in the repo, and set it as the **default value of `$rsh_json` directly in MobileCoach** — then `initDefaultState()` (and the `'0'` = first-run branch) can shrink or disappear, since the first run would already find valid JSON in `$rsh_json`.

- **Today:** the default state stays embedded in `defaultStateTemplate()` inside the script.
- **Open design points:** the move introduces a sync hazard — whenever the JSON file changes, the `$rsh_json` default value in MobileCoach must be updated by hand, which is easy to forget. Needs a guard (e.g. a checklist item, a test, or a hook — see the version-stamp item below) so a data change can't silently diverge from what MobileCoach seeds. Also unclear how deployed participants (whose `$rsh_json` already holds state) are affected by a changed default — probably not at all, but worth confirming.
- **Done when:** the JSON is extracted to its own file, the MobileCoach default value is wired, the don't-forget-to-sync mechanism is decided, the script is measurably lighter for MobileCoach (or the measurement says it wasn't the size), and the outcome is logged via `/log-decision`.

## Lighten the script: split the code by concern into separate scripts

Second step of the performance effort (see the item above): instead of loading the whole 600+ line script for every command, split it into several scripts by concern, so MobileCoach only loads the code an action actually needs (e.g. `enter('aXy')` doesn't need menu population or progress code).

- **Today:** everything stays in one `src/ReactStateHelper.js`; every MobileCoach rule that runs a command carries the full script.
- **Open design points:** this reverses decision #1 (single self-contained script) and would require relaxing the self-containedness checks in `test/MobileCoachPlatformConstraints.test.js` — it needs its own decision entry. The split boundaries are unclear (per command? shared core classes + per-command wrappers?), and any shared code copied into multiple scripts creates a serious sync hazard: an update must be pasted into **several** places inside MobileCoach, and forgetting one means silently divergent behavior. Needs a guard mechanism analogous to the wrapper-variable checks (e.g. generated artifacts from a single source, a checksum comment per script, or a test that rebuilds and compares the deployed set — see the version-stamp item below). Also unverified whether script size is actually what's slowing MobileCoach down — worth measuring before paying the complexity cost.
- **Done when:** the time sink is measured, split boundaries and a keep-in-sync mechanism are chosen, decision #1 is superseded via `/log-decision`, and the constraint tests enforce the new shape.

## Version-stamp the deployed artifacts and detect drift at load time

Both "lighten" items above multiply the copy-paste surfaces (several scripts, plus the default state JSON as a MobileCoach default value), and both flag the same hazard: updating one place in MobileCoach and forgetting another causes silently divergent behavior. Idea: embed a shared version marker in every deployed artifact — a comment line in each script, a field in the JSON (e.g. `"version"`) — and have the script check at load time that its own marker matches the JSON's, raising an error via `$rsh_error` on mismatch instead of running with mixed versions.

- **Today:** one script with the JSON embedded in it — drift between deployed parts is impossible by construction, so nothing is stamped or checked.
- **Open design points:**
  - **What the marker is.** A raw commit id has a chicken-and-egg problem: a commit's own hash cannot be embedded in the files of that commit, because the hash depends on the file contents. And stamping each file with the id of the commit that last touched it breaks anyway — a file untouched by a change keeps its old stamp, so an honestly-deployed set would false-alarm. The marker therefore has to be **one shared value across all artifacts**, and the candidates are: a hand-bumped version constant living in one repo file and copied into every artifact (a test can enforce agreement across the repo files, analogous to the wrapper-variable checks), or a stamp written at **export time** — a small "prepare for MobileCoach paste" npm script that writes the current HEAD commit id into every artifact just before deployment, so the stamp identifies the deployment rather than the commit.
  - **Which comparisons the runtime check covers.** Drift between files *inside the repo* needs no runtime check — a test compares stamps for free. The valuable runtime check is repo ↔ MobileCoach ("one paste forgotten"): each pasted script compares its embedded stamp against the JSON's. But `$rsh_json` persists per participant, so after any script update every existing participant's stored JSON still carries the old stamp — a strict equality check at load would hard-fail all existing participants on every deploy. Options: check only against the first-run default JSON, treat a stamp mismatch in persisted state as "migration needed" rather than as an error, or keep the stamp out of participant state entirely (e.g. a separate `$rsh_stateVersion` MobileCoach variable seeded alongside the JSON default). In the split-script world, script ↔ script drift also needs a channel — e.g. each script writes its stamp into a shared variable and the wrapper compares.
  - **MobileCoach constraint:** the stamp comment must contain no `$` (the save-time scan) — plain version strings and hex commit ids are fine.
- **Done when:** the marker format and stamping mechanism are chosen, the runtime check (including its story for already-deployed participant state) is implemented with tests, it satisfies the "needs a guard" points of the two lighten items above, and the decision is logged via `/log-decision`.

## Batch commands: several `;`-separated commands in one `$rsh_cmd`

Currently every command costs a full script execution in MobileCoach. Allowing `$rsh_cmd = "enter('aXy'); completeActivity()"` — the wrapper splits on `;` and executes the parts one after another in a single run — would cut round-trips and complements the performance items above.

- **Today:** one command per run. (Note: naive `;`-chaining does *not* work — the wrapper evals `'helper.' + $rsh_cmd`, so only the first part gets the `helper.` prefix.)
- **Open design points:** semantics need deciding. `$rsh_result` first: in practice it is never meaningfully populated — MobileCoach flows only run doer commands (enter, complete, populate), which all return nothing, so `$rsh_result` is always `''`; the query commands the developer guide routes through it aren't actually used from MobileCoach. Options: **remove `$rsh_result` entirely** (drop it from the wrapper, the variable table, the MobileCoach declarations, and rewrite the developer guide's query-command section — queries would then need another output channel, e.g. the direct-write model of the item below), or **turn it into a batch report**: a comma-separated list with one entry per executed command, each pairing the command with its result. Then the error semantics (does a middle command throwing stop the batch? is state from already-executed parts saved or rolled back?) and whether a plain `split(';')` is safe (currently yes — arguments are ids, letters and numbers only — but that assumption should be pinned in a test).
- **Done when:** the result/error/partial-state semantics are decided, the split-and-iterate loop is implemented with tests, the syntax is documented in the content editor guide, and the decision is logged via `/log-decision`.

## Invert the output flow: pre-initialise all variables to null, commands write results directly

Today the wrapper assembles the whole output object at the end of every run, pulling values out of the helper via getters — which forces guards like the `getProgressAdvice()` try/catch (decision #37): a getter that throws must not take the whole output object down with it. Proposed inversion: initialise **all** MobileCoach output variables to `null` at the very start, then let each command write its own results directly where they are computed (e.g. `populateMenuForModules()` writes the menu label/id variables itself). Untouched variables simply stay `null`; the guards disappear.

- **Today:** end-of-run output object with per-getter guards.
- **Open design points:** it changes the output contract. Today every run deliberately (re)writes every variable — that's how menu slots reset to `''` after non-populate commands so no stale menu survives. With null-defaults that behavior needs re-deciding: decision #37 already records that a key *omitted* from the output object makes MobileCoach **keep the variable's previous value** (that trap is exactly why the always-write model exists), but what an explicit `null` value does (keep? clear? store the string `"null"`?) is unverified — needs a live check and a field note. The platform itself will not help: MobileCoach [never cleans up variables](mobilecoach-field-notes.md#variables-are-never-cleaned-up-stale-values-survive-forever), so any variable the script stops overwriting stays stale forever. The always-on outputs (`$rsh_progressAdvice` per decision #37, `participantGroup` with its merged location + completion snapshot) need a home if there's no end-of-run assembly. And commands writing outputs directly need a seam the Node tests can observe (an output collector object rather than true globals). Interacts with the batch-commands item above: with several commands per run, "each command writes its own outputs" is the natural model, but later commands may overwrite earlier ones' outputs.
- **Done when:** MobileCoach's null-write behavior is tested live, the new contract per variable (always written vs. command-owned) is decided, wrapper + commands are refactored with tests, and decision #37's guard mechanism is superseded via `/log-decision`.

## Check the MobileCoach flow export (HTML) into the repo

MobileCoach can export an HTML table of all flow elements. Plan: check that export into the repo so (a) Claude has the real flow structure — dialogs, rule sequencing, cascade/jump targets, variable writes, if/else trees — as context when supporting design decisions, and (b) each time a newer export is committed, the diff against the previous version can be summarized in flow terms ("what changed since the last export").

- **Today:** knowledge about the deployed flow lives only in heads and in the [field notes](mobilecoach-field-notes.md); Claude reasons about MobileCoach blind.
- **Open design points:** whether the HTML is diff-friendly as-is or needs normalizing on check-in (stable element ordering, stripped timestamps/volatile ids — otherwise every diff is noise); whether it belongs in `docs/` or a dedicated directory, and whether GitHub Pages would then publish it (may or may not be wanted); whether the export contains anything that shouldn't be public in the repo; and whether a small parse/summarize step is needed for the "what changed" reports or the raw diff suffices. Alongside the export, a fuller field-note write-up of dialog internals (rules, cascading vs. jumping — [stub exists](mobilecoach-field-notes.md#moving-between-dialogs-cascading-vs-jumping) — variable writes, if/else trees) is planned.
- **Done when:** a first export is produced, location/normalization is decided, it is committed, and one real "what changed" round-trip on a second export has validated the workflow.

## Helper dialogs: outsource `enter(…)` into shared black-box dialogs

Today every module/session/activity dialog issues its own `enter('id')` with a hardcoded id. Plan: shared helper dialogs (enterModule / enterSession / enterActivity) that content editors simply **cascade** to at the start of their dialog — no id needed, because after a menu tap `$participantNextMicroDialogIdentifier` already holds the current dialog's id, so the helper runs `enter($participantNextMicroDialogIdentifier)`. This is the general direction for MobileCoach-side architecture: complexities hidden behind generally available dialogs that work like helper functions — content editors call the black box, never see the internals.

- **Today:** each content dialog contains its own full `enter('id')` command setup.
- **Open design points:** two need care. **(1) Staleness — the entry-path interface:** the implicit argument relies on `$participantNextMicroDialogIdentifier` being current, but MobileCoach [never cleans up variables](mobilecoach-field-notes.md#variables-are-never-cleaned-up-stale-values-survive-forever) — a content dialog reached any way other than a menu tap would make the helper enter whatever was tapped last. Direct jumps from inside the app shouldn't be possible, but **push notifications** ("you could do X again") are a real non-menu entry path. Proposed interface: the invariant is *"whatever leads into a content dialog must set `$participantNextMicroDialogIdentifier` first"* — menu taps do this natively; push notifications point to a **launcher helper dialog** that first sets `$participantNextMicroDialogIdentifier` and then jumps to the target dialog, mirroring the menu-tap contract. Sub-question: can the launcher's jump target be taken from the variable itself (the way menu routing navigates), so one generic launcher suffices — or must the jump be hardcoded, requiring one launcher per push target? And the guard for when the invariant is broken anyway: if `$participantNextMicroDialogIdentifier` is not set at all (e.g. a DEBUGGER coach jumping straight into an activity — the variable would hold MobileCoach's uninitialised `0` or a stale id), the enter-helper should surface an **error** to the user rather than silently enter the wrong thing; and if the participant is a DEBUGGER, additionally offer an input dialog to type an id manually and proceed. These guard messages are DEBUGGER-facing, so the shared banner convention (decision #48) applies — still open whether the script interpolates `'$debugBanner'` itself (in which case it should also guard against a banner left at MobileCoach's default `'0'` by raising via `$rsh_error` instead of rendering debug messages with a bare `0`) or leaves prepending entirely to the flow side. Detection limit: `0` catches only the never-tapped-anything case — after any earlier tap the variable holds a plausible-looking stale id; the per-level helpers can at least cross-check the id's level letter (enterActivity rejects a non-`a…` id), which catches cross-level staleness but not a stale id of the right level. **(2) One helper or three — leaning: three.** All three levels do the same thing today, and a single enter-dialog could technically branch on the id's level letter (`m`/`s`/`a`); but three dialogs force content editors to state *what kind of thing* they are entering — too much hiding of logic invites carelessness. The black-box principle hides the *internals*, not the *intent*. (Also, per-level behavior is expected to diverge later anyway.) Also worth verifying the cascade return-chain behaves as assumed (see the [unverified jump-inside-cascade note](mobilecoach-field-notes.md#moving-between-dialogs-cascading-vs-jumping)).
- **Done when:** the helper dialogs are built in a debug coach, the non-menu-tap entry path (push launcher, DEBUGGER guard) is tested, the pattern is documented in the content editor guide, and the decision is logged via `/log-decision`.
